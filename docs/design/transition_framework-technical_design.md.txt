# Battlecraps Transition Framework ‚Äî Technical Design

---

## Guiding Principles

1. **One owner per concern.** The `TransitionOrchestrator` owns all transition logic. `App.tsx` routing defers to it entirely.
2. **Configuration over code for floors.** Adding a new floor means editing config, not touching component logic.
3. **Display state vs. logical state.** The store carries two layers: what the server knows, and what the player sees. The orchestrator controls when they sync.
4. **Phases are atomic and sequential.** No transition phase starts until the previous one formally completes.
5. **floors.md is the contract.** It defines the schema, documents all existing floors, and is the handoff document for developers building new ones.

---

## System Architecture

```
TransitionOrchestrator (React Context)
  ‚îÇ
  ‚îú‚îÄ‚îÄ TransitionRegistry         maps event type ‚Üí ordered phase sequence
  ‚îú‚îÄ‚îÄ PhasePlayer                renders current phase, owns timers & gates
  ‚îú‚îÄ‚îÄ GameStateSnapshot          frozen pre-transition state (fixes race condition)
  ‚îî‚îÄ‚îÄ phases/
        ‚îú‚îÄ‚îÄ TitleScreenPhase
        ‚îú‚îÄ‚îÄ MarkerCelebrationPhase
        ‚îú‚îÄ‚îÄ MarkerIntroPhase
        ‚îú‚îÄ‚îÄ FloorRevealPhase       ‚Üê new, floor-config-driven
        ‚îú‚îÄ‚îÄ BossEntryPhase         ‚Üê refactored
        ‚îú‚îÄ‚îÄ BossVictoryPhase       ‚Üê refactored
        ‚îú‚îÄ‚îÄ VictoryCinematicPhase  ‚Üê new, replaces win path
        ‚îî‚îÄ‚îÄ GameOverPhase          ‚Üê refactored
```

The `App.tsx` routing collapses from ~60 lines of interlocked boolean gates to a single render:

```tsx
<TransitionOrchestrator>
  <GameCanvas />   {/* TableBoard ‚Äî only mounts when orchestrator is IDLE */}
</TransitionOrchestrator>
```

## Floor Configuration System

### The Schema (`packages/shared/src/floors.ts`)

```typescript
export interface FloorConfig {
  id: number;                    // 1-indexed, sequential
  name: string;                  // e.g. "The Riverboat"
  tagline: string;               // One atmospheric line for the reveal screen
  introLines: string[];          // 2-3 lines shown during floor reveal
  bossTeaser: string;            // Shown at bottom of reveal: "Mme. Le Prix awaits‚Ä¶"
  atmosphere: 'gritty' | 'elegant' | 'electric' | string;  // extensible
  theme: FloorTheme;             // Visual tokens (existing system, extended)
  markers: MarkerConfig[];       // Exactly 3 per floor
}

export interface MarkerConfig {
  targetCents: number;
  venue: string;
  isBoss: boolean;
  boss?: BossConfig;
}

export interface BossConfig {
  name: string;
  title: string;
  rule: BossRule;
  ruleIcon: string;              // emoji or icon key for visual display
  flavorText: string;            // Shown in BossEntryPhase
  dreadLine: string;             // Short ominous line before CTA appears
  comp: CompConfig;
}

export interface CompConfig {
  id: string;
  name: string;                  // "The Member's Jacket"
  description: string;           // What it does
  icon: string;                  // emoji or icon key
}
```

### floors.md ‚Äî Structure

The document has two purposes: **developer reference** and **designer handoff**. Its sections:

```
1. Overview ‚Äî what floors are, how they map to the gauntlet
2. Floor Configuration Schema ‚Äî the TypeScript interface, field-by-field explanation
3. How to Add a New Floor ‚Äî step-by-step checklist (5 steps, no code surgery)
4. How to Add a New Boss Rule ‚Äî step-by-step checklist
5. Floor Definitions
   ‚îî‚îÄ‚îÄ Floor 1: VFW Hall
   ‚îî‚îÄ‚îÄ Floor 2: The Riverboat
   ‚îî‚îÄ‚îÄ Floor 3: The Strip
6. Visual Theming Guide ‚Äî how to design FloorTheme tokens for a new floor
7. Boss Rule Catalogue ‚Äî all existing rules, their icons, their mechanical description
```

The document is **normative** ‚Äî it is the definition of what the game contains. Code reads from `floors.ts` which is populated from this document. A designer proposes in `floors.md`, a developer translates to `floors.ts`. No other files need touching for new floor content.

## The TransitionOrchestrator

### Core Types

```typescript
type TransitionType =
  | 'TITLE'
  | 'MARKER_CLEAR'
  | 'MARKER_INTRO'
  | 'FLOOR_REVEAL'
  | 'BOSS_ENTRY'
  | 'BOSS_VICTORY'
  | 'VICTORY'
  | 'GAME_OVER';

type PhaseAdvanceMode = 'auto' | 'gated' | 'animation';
// auto      ‚Äî advances after `duration` ms
// gated     ‚Äî waits for player tap (shows CTA)
// animation ‚Äî waits for a named animation to fire onComplete()

interface TransitionPhase {
  id: string;
  advanceMode: PhaseAdvanceMode;
  duration?: number;             // only for 'auto'
  animationKey?: string;         // only for 'animation'
  component: string;             // key into PhaseComponentRegistry
}

interface TransitionData {
  // Populated per-transition, only the fields each phase needs
  markerIndex?: number;
  markerTarget?: number;
  floorId?: number;
  isFloorBoundary?: boolean;
  bossName?: string;
  compEarned?: CompConfig;
  finalBankroll?: number;
  maxSingleRollWin?: number;
  markersCleared?: number;
  // ‚Ä¶ extensible without schema changes
}
```

### The Registry

```typescript

  MARKER_CLEAR: [

    { id: 'modal',       advanceMode: 'gated',                                       component: 'MarkerCelebrationModal' },
  ],

  FLOOR_REVEAL: [

  ],

  VICTORY: [
    { id: 'explosion',   advanceMode: 'auto',  duration: 2500, component: 'VictoryExplosionPhase' },
    { id: 'recap',       advanceMode: 'gated',                 component: 'VictoryRecapPhase' },
    { id: 'sendoff',     advanceMode: 'gated',                 component: 'VictorySendoffPhase' },
  ],

    { id: 'over',        advanceMode: 'gated',                 component: 'GameOverPhase' },
  ],
};
```

Adding a new transition type: add one entry here, one component file, zero changes elsewhere.

## State Snapshot System (Race Condition Fix)

New fields in the Zustand store:

```typescript
// In useGameStore.ts
celebrationSnapshot: {
  markerIndex: number;      // The marker being celebrated (OLD value)
  targetCents: number;      // Its target
  floorId: number;          // Its floor
  bankrollBefore: number;   // Bankroll at moment of hit
  bankrollAfter: number;    // New bankroll (for the delta display)
} | null;

activeTransition: TransitionType | null;
transitionData: TransitionData;
transitionPhaseIndex: number;
```

**How it works:**

When `applyPendingSettlement()` detects a TRANSITION result:
1. Snapshot the **current** (old) marker state into `celebrationSnapshot`
2. Set `activeTransition: 'MARKER_CLEAR'`
3. Do **not** update `currentMarkerIndex` in the display layer ‚Äî the logical store has it, but `celebrationSnapshot.markerIndex` is what every UI component reads during celebration
4. The `ChipRain` component fires its existing `onComplete` callback ‚Üí orchestrator advances to the `modal` phase
5. Modal cleared ‚Üí `celebrationSnapshot` cleared ‚Üí `currentMarkerIndex` is now visible in UI

The player never sees the new state until the orchestrator hands it to them.

## Maximum Win Tracking

**Server side** (`apps/api/src/routes/rolls.ts`):
- Track `maxSingleRollWinCents` on the run record
- After every roll, if `bankrollDelta > maxSingleRollWinCents`, update it
- Include in `turn:settled` payload

**Client side** (`useGameStore.ts`):
- `maxSingleRollWin: number` in store
- Updated from every `turn:settled` event
- Surfaced in `VictoryRecapPhase`

## Full Transition Flow ‚Äî Complete Sequence

```
Game Start (first load)
  ‚Üí TitleScreenPhase (gated) ‚Üí bootstrap() ‚Üí TableBoard

Roll ‚Üí Marker Clear
  ‚Üí [snapshot frozen]
  ‚Üí ChipRain + Payout Pops + Screen Flash (against old state)
  ‚Üí chipRainComplete fires ‚Üí MarkerCelebrationModal (gated)
  ‚Üí [snapshot cleared, new marker visible]
  ‚Üí PubScreen
  ‚Üí If floor boundary:
      ‚Üí FloorRevealPhase (2.5s auto) ‚Üí FloorRevealConfirmPhase (gated)
  ‚Üí If boss marker:
      ‚Üí BossEntryDreadPhase (1.8s auto) ‚Üí BossEntryRevealPhase (gated)
  ‚Üí MarkerIntroPhase (2.5s auto)
  ‚Üí TableBoard (first roll ready)

Roll ‚Üí Boss Clear
  ‚Üí [same chip rain sequence]
  ‚Üí VictoryExplosionPhase (auto) ‚Üí BossVictoryCompPhase (gated)
  ‚Üí PubScreen
  ‚Üí [floor reveal if applicable]
  ‚Üí MarkerIntroPhase
  ‚Üí TableBoard

Roll ‚Üí Final Marker Clear (Victory)
  ‚Üí [chip rain]
  ‚Üí VictoryExplosionPhase (2.5s auto)
  ‚Üí VictoryRecapPhase (gated ‚Äî markers, crew, max roll, bankroll)
  ‚Üí VictorySendoffPhase (gated ‚Äî flavor text + play again)

Shooters = 0
  ‚Üí GameOverPhase (gated ‚Äî progress + tone-calibrated message + play again)
  ‚Üí Play Again ‚Üí bootstrap(true) ‚Üí TableBoard (no title screen)
```

## Implementation Phases

Each phase is a discrete, independently shippable unit. No phase requires future phases to function. Each one improves the game as it lands.

---

### Phase 1 ‚Äî Foundation: TransitionOrchestrator + Snapshot System
**Delivers:** The race condition fix. Zero visual change to existing modals.

- Create `packages/shared/src/floors.ts` with all type definitions
- Add `celebrationSnapshot`, `activeTransition`, `transitionData`, `transitionPhaseIndex` to Zustand store
- Build `TransitionOrchestrator` context and `PhasePlayer` component (thin shell ‚Äî just wires timers and gates)
- Build the `TRANSITION_REGISTRY` (references component names as strings ‚Äî components don't need to exist yet)
- Refactor `applyPendingSettlement()` to populate snapshot and set `activeTransition`
- Refactor `App.tsx` to render `<TransitionOrchestrator>` ‚Äî existing modals wired as phase components (BossEntryModal, MarkerCelebration, etc. unchanged, just registered)
- Remove `pubReady` and `bossEntryAcknowledged` boolean gates ‚Äî orchestrator owns this now

**Test:** Marker clear no longer shows new marker state during chip rain. All existing modals still appear. Nothing visually broken.

---

### Phase 2 ‚Äî Floor Configuration System
**Delivers:** `floors.md` and `floors.ts` as living documents. Existing content migrated.

- Create `docs/floors.md` with full schema documentation and all 3 floors defined
- Create `packages/shared/src/floors.ts` with `FloorConfig[]` array containing VFW Hall, The Riverboat, The Strip
- Migrate `packages/shared/src/config.ts` gauntlet data to derive from `floors.ts` (GAUNTLET array becomes a computed flatten of `FLOORS.flatMap(f => f.markers)`)
- Extend `lib/floorThemes.ts` to derive tokens from `FloorConfig.theme`
- No visual change ‚Äî this is a data migration

**Test:** All existing floor-aware components (BossRoomHeader, PubScreen, floorThemes) still work identically.

---

### Phase 3 ‚Äî Marker Celebration + Intro
**Delivers:** Clean marker clear sequence. MarkerIntro card after pub.

- Refactor `MarkerCelebration` into `MarkerCelebrationPhase` ‚Äî full-screen takeover, reads from `celebrationSnapshot`
- Add `chipRainComplete` callback to `ChipRain` ‚Äî fires `advancePhase()` on orchestrator
- Build `MarkerIntroPhase` ‚Äî slides in after pub, shows floor pip progress + next target
- Wire both into the registry

**Test:** Marker clear feels sequential and clean. Player sees old state during celebration, new state revealed by the modal. Marker intro orients the player before first roll.

---

### Phase 4 ‚Äî Floor Reveal
**Delivers:** Floors feel like acts. New venue announced with weight.

- Build `FloorRevealPhase` ‚Äî full-screen wipe, reads `FloorConfig.name`, `FloorConfig.introLines`, `FloorConfig.bossTeaser`
- Build `FloorRevealConfirmPhase` ‚Äî player tap CTA
- Detect floor boundary in orchestrator: `Math.floor(newMarkerIndex / 3) > Math.floor(oldMarkerIndex / 3)`
- Insert `FLOOR_REVEAL` transition into sequence at the right moment (after pub, before boss entry)

**Test:** Clearing marker 2 ‚Üí Floor Reveal for "The Riverboat" ‚Üí Boss Entry for Mme. Le Prix. Adding Floor 4 in the future: add one entry to `FLOORS` array in `floors.ts`, zero component changes.

---

### Phase 5 ‚Äî Boss Sequence Polish
**Delivers:** Boss entry has dread and weight. Boss victory has ceremony.

- Build `BossEntryDreadPhase` ‚Äî ominous auto phase, CTA button hidden for 1.8s
- Refactor `BossEntryRevealPhase` from existing `BossEntryModal` ‚Äî adds rule icon, richer CTA copy from `BossConfig`
- Build `BossVictoryPhase` ‚Äî triumph moment
- Build `BossVictoryCompPhase` ‚Äî animated comp reveal, reads from `CompConfig`

**Test:** Boss entry feels like walking into a boss room. Victory feels earned.

---

### Phase 6 ‚Äî Title Screen
**Delivers:** First impression cinematic.

- Build `TitleScreenPhase` ‚Äî felt wipe in, game name, floor/boss teaser icons, single CTA
- Add `hasSeenTitleScreen` to local storage (not store) ‚Äî persists across sessions
- `App.tsx`: if `!hasSeenTitleScreen`, push `TITLE` transition before bootstrap; set flag on proceed
- `Play Again` path: calls `bootstrap(true)` directly, skips TITLE

**Test:** Fresh load shows title. Play Again skips it. Hard refresh shows it again.

---

### Phase 7 ‚Äî Maximum Win Tracking
**Delivers:** Data infrastructure for win screen and future unlocks.

- Add `maxSingleRollWinCents` column to run schema
- Track in `rolls.ts` after every settlement
- Include in `turn:settled` payload
- Add `maxSingleRollWin` to store, updated on every settlement
- No UI yet ‚Äî this is pure data prep for Phase 8

**Test:** After a run, `maxSingleRollWin` in store reflects the actual highest single-roll payout.

---

### Phase 8 ‚Äî Victory Cinematic
**Delivers:** The win screen overhaul. Epic ending.

- Build `VictoryExplosionPhase` ‚Äî Torrent ChipRain from all edges, "GAUNTLET CLEARED" headline, screen shake
- Build `VictoryRecapPhase` ‚Äî floor-by-floor with checkmarks, bosses defeated, final crew, bankroll peak, max single roll win (animated counter)
- Build `VictorySendoffPhase` ‚Äî flavor text keyed to final bankroll tier, "PLAY AGAIN"
- Detect VICTORY vs. GAME_OVER in orchestrator: `markersCleared >= 9`
- Route to `VICTORY` sequence instead of `GAME_OVER`

**Test:** Clearing marker 8 triggers the cinematic. Game over on shooter loss shows `GAME_OVER` as before.

---

### Phase 9 ‚Äî Game Over Polish
**Delivers:** Progress acknowledgment, tone calibration.

- Add pip progress display to `GameOverPhase` (how many of 9 markers cleared)
- Add tone calibration: early loss (0-3) vs. mid (4-6) vs. near miss (7-8) ‚Äî different headline, flavor text
- No structural changes ‚Äî enhancement to existing component

## What "Adding a New Floor" Looks Like After This

A developer receives a `floors.md` entry with the new floor filled in. Their checklist:

1. Add `FloorConfig` entry to `FLOORS` array in `floors.ts`
2. Add `FloorTheme` tokens (colors, felt, glow) to the theme section
3. Add boss comp logic to `recruit.ts` if the comp type is new (most won't be)
4. Add boss rule logic to `rolls.ts` if the rule type is new (most won't be)
5. Done ‚Äî floor reveal, marker intro, boss entry, boss victory all pick up the config automatically

No component surgery. No new modal files. No routing changes.
